# AP 64KB Timeout Trigger Evidence - 2026-06-01

## Scope

This note continues the same line: why the AP firmware 64KB path is broken. It does not propose bypassing the 64KB path with forced 4KB upload semantics.

No live device access, firmware execution, DLL deployment, service control, or I2C transaction was performed.

`FUN_0000C370` is now confirmed to be a watchdog/service write:

```asm
0000c370  ldr r1,[0x0000c37c]   ; 0x40068000

0000c374  str r0,[r1,#0x0]

```

Meaning: `BA44` is not sleeping inside the poll loop. It repeatedly feeds a watchdog/service register and reads SPI status. The timeout counter at `0x200000FE` must be decremented elsewhere, most likely by an interrupt/timer path.

### 1. The AP status-poll helper uses one fixed timeout for erase and program

`FUN_0000BA44`:

```asm
0000ba4a  mov.w r0,#0x12c
0000ba4e  strh  r0,[r5,#0x0]
...
0000ba66  movs  r0,#0x5
0000ba68  bl    FUN_0000B990
...
0000ba6c  bl    FUN_0000C370
0000ba72  bl    FUN_0000B990
...
0000ba7c  cmp   timeout,#0
0000ba80  movs  r0,#0
...
0000ba9e  movs  r0,#1
```

Decompile shape:

```c
*timeout = 300;
FUN_0000B990(0x05);        // read status register
do {
    FUN_0000C370();
    status = FUN_0000B990(0);
    if (*timeout < 1) return 0;
} while ((status & 1) != 0);
return 1;
```

The same `300` budget is used after:

- 4KB erase `0x20`
- 64KB erase `0xD8`
- page program `0x02`

That is suspicious because these operations have very different expected durations.

Extra cross-reference result:

```text
0x200000FE references:
  BA44 writes 300 and reads it during SPI WIP polling.
  B760 and B9C4 also use the same counter pattern for other wait loops.
```

The dump did not expose a direct ordinary-code decrement site for `0x200000FE`. That makes the tick duration unresolved, but it also means extending the wait by inserting a simple busy delay around `C370` would be the wrong mental model. The firmware uses a shared async timeout counter, not an inline decrement loop.

### 2. Typical SPI NOR timing makes a short 64KB timeout plausible

Public SPI NOR flash datasheets show that 64KB block erase can be much slower than page program and often slower than 4KB sector erase:

- Winbond W25Q128FV lists page program at 0.7 ms typical / 3 ms max, 4KB sector erase at 100 ms typical / 400 ms max, and 64KB block erase at 150 ms typical / 2000 ms max. Source: https://www.buydisplay.com/download/manual/W25Q128FV_Datasheet.pdf
- Macronix MX25L12835F lists page program at 0.5 ms typical / 1.5 ms max, 4KB sector erase at 30 ms typical / 120 ms max, and 64KB block erase at 280 ms typical / 650 ms max. Source: https://www.digikey.com/en/htmldatasheets/production/2014243/0/0/1/mx25l12835f
- Macronix MX25L12873F lists 4KB sector erase at 30 ms typical / 120 ms max and 64KB block erase at 0.28 s typical / 0.65 s max. Source: https://www.digikey.com/en/htmldatasheets/production/2702709/0/0/1/mx25l12873f

The exact panel flash chip is not proven yet, but these are representative SPI NOR timings. The important engineering point is the ratio:

```text
page program: usually ~1 ms
4KB erase: usually tens to hundreds of ms
64KB erase: often hundreds of ms, sometimes up to seconds
```

A single fixed `300` poll budget can easily be enough for page program and maybe 4KB erase, while being marginal or too short for 64KB erase depending on the AP timer tick.

### 3. 64KB helper still hides this timeout

The timeout hypothesis only matters because `FUN_0000B4D0` discards the result:

```asm
0000b530  bl    FUN_0000BA44
0000b534  movs  r0,#1
```

Therefore if 64KB erase times out, AP firmware converts that failure into success.

### 4. 4KB helper uses the same timeout but does not hide failure

`FUN_0000B910`:

```asm
0000b970  bl    FUN_0000BA44
0000b974  cmp   r0,#0
0000b976  beq   fail
0000b978  movs  r0,#1
```

That means the AP code authors already knew the poll can fail. The bug is not the existence of a poll. The bug is that the 64KB and page-program helpers ignore the poll result.

## Status / Protection Re-check

The AP exports do not currently prove a robust status/protection setup path:

- WREN helper `FUN_0000BAB4` sends opcode `0x06`.
- It does not read status afterward to verify WEL.
- The only confirmed status read in the erase/program helpers is opcode `0x05` for WIP polling.
- No direct SPI transfer of common status/protection opcodes `0x35`, `0x15`, `0x01`, `0x31`, `0x11`, or `0x50` has been proven from the current exports.

Important correction:

`0x9F` appears in the AP dump, but current evidence shows it as a flash-content magic check after reading address `0x01400000` through normal read opcode `0x03`. It is not proven to be a JEDEC-ID SPI command.

## Current Root Cause Ranking

### Rank 1: Short or insufficient 64KB erase poll timeout plus ignored failure

This is now the strongest complete explanation:

```text
0xD8 erase starts
WIP remains set beyond AP's 300-tick budget
BA44 returns 0
B4D0 overwrites the return to 1
upload loop advances to next block / program phase
flash region after the first 64KB boundary is not correctly erased
B6CC also hides program failures
host sees success
panel displays partially stale/corrupt data
```

This explains the static image boundary perfectly and explains why the static 4KB route works.

### Rank 2: WREN/WEL not verified

The AP sends WREN but does not verify WEL. If WREN occasionally fails or is cleared unexpectedly, erase/program can be rejected while AP still proceeds.

This could be a contributing factor, but it is less directly tied to the 64KB-only symptom than the timeout/ignored-return chain.

### Rank 3: Block protection/status register state

Datasheets state block erase can be rejected when protection bits cover the block. The current AP exports do not show a clear protection-register management path, so this remains possible but not proven.

### Rank 4: Missing 4-byte address mode

Weak. AP upload initialization sends `0xB7`, so 4-byte address mode appears intentional and present.

## Repair Direction, Still Focused On 64KB

The clean 64KB repair should not change GIF semantics, destination, payload, or host F1 layout.

The direct AP-side repair concept is:

1. Make `FUN_0000B4D0` propagate `FUN_0000BA44` failure.
2. Increase the poll budget for 64KB erase, ideally only for the 64KB helper rather than globally.
3. Make `FUN_0000B6CC` propagate page-program poll failure.
4. Consider WEL verification after WREN only after the timeout/return-propagation fix is understood.

The best first offline patch design is therefore not â€œforce GIF to 4KB.â€ It is:

```text
64KB erase: issue 0xD8, poll long enough for real 64KB erase, return failure if WIP never clears.
page program: return failure if WIP never clears.
```

## What Still Needs Proof

We still need one missing piece before claiming full physical root cause:

```text
How long is one AP timeout tick?
```

If 300 ticks is materially below common 64KB erase max time, then the physical root cause becomes very strong:

```text
AP timeout too short for 64KB erase; AP then ignores the timeout.
```

If 300 ticks is already many seconds, then the likely root cause shifts toward WREN/WEL or protection rejection. Either way, return propagation is still required because the existing firmware silently corrupts state after failure.

## Updated Repair Implication After C370 Dump

Because `C370` is only a watchdog/feed helper, the most natural repair is not to add arbitrary delay inside the poll loop. The better repair remains:

```text
give 64KB erase a larger timeout counter value before polling,
then propagate BA44's real result.
```

In practical patch-design terms, a direct 64KB repair would likely modify `B4D0` around the `BA44` call:

```text
before/inside 64KB wait: use a 64KB-appropriate timeout budget
after BA44 returns: do not overwrite failure with success
```

A global `BA44` timeout increase is easier but less precise, because it also changes page-program and 4KB behavior. A 64KB-specific timeout is cleaner, but requires more careful patch-space planning.
