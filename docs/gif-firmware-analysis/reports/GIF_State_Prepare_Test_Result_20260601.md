```text
[ 2026-06-01 01:05:56 ] ucVga.GvLcdExApi  ERROE  GetFWVersion ReadData fail
[ 2026-06-01 01:07:18 ] ucVga.I2CApi  ERROE  SendData GvWriteI2C fail
[ 2026-06-01 01:07:18 ] ucVga.GvLcdApi  ERROE  SendImage SendData Image Byte fail
```

Important contradiction:

- GCC logged a payload write failure during upload at `01:07:18`.
- The outer `SendImage` still returned `True` at `01:07:59`.

The IL explains this. In the 256-byte page loop, a failed page write logs:

```text
SendImage SendData Image Byte fail
```

Then it retries internally. It only returns false after the retry counter passes its failure threshold. Therefore `SendImage=True` means the retry loop eventually completed, not that the panel reliably accepted every write on the first try or reached a healthy playback state.

This repeats the earlier static lesson:

> `SendImage=True` is not strong evidence that the panel-side custom storage/playback state is healthy.

## Result

The first GIF hypothesis is weakened:

> Simply preparing intended GIF display/template state before upload does not prevent corruption for this GIF.

The remaining strongest explanations are:

1. The large GIF `chunk_mode=2` upload path is unstable or has panel-side timing/finalization sensitivity.
2. This specific GIF profile is unsafe: 60 ms delay, full 320x170 frames, 1.6 MB payload.
3. A retry-successful upload can still leave panel storage/playback state corrupt.
4. The panel may need a smaller/simpler GIF that naturally uses `chunk_mode=1`, not a forced `0x02 -> 0x01` mutation on a large payload.

## Recommended Next Step

Do not immediately force the same 1.6 MB GIF through `chunk_mode=1`.

The safer next experiment should be designed as:

```text
GIF state preparation remains enabled:
  OpenLcd(true)
  SetDisplay(0,1)
  SetMode(5)
  SetImageTpl(GIF,true)

Use a deliberately tiny/simple GIF:
  2-3 frames
  delay >= 100 ms
  mostly solid/simple pixels for strong RLE compression
  target converted animation.bin ideally < 20480 bytes

Let GCC naturally choose F1[0x11] == 1 because payload < 20480.
Do not force F1[0x11].
Do not change target/storage.
Do not use SetLoop/F3 yet.
Do not use Save/AA yet.
```

Why this is better:

- It tests the user's second hypothesis, `0x02 path may be the problem`, without lying to the protocol.
- If payload is genuinely small, `chunk_mode=1` is selected naturally by GCC's own logic.
- It avoids applying `chunk_mode=1` to a large payload that public protocol expects to use `chunk_mode=2`.

Only after that should a forced `0x01` large-GIF DLL be considered.
